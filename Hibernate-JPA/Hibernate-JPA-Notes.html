What Spring Data JPA Actually Is

What it does: a data-access abstraction layer built on top of JPA and Hibernate that eliminates repository boilerplate — you write an interface, Spring generates the implementation at runtime via a dynamic proxy. CRUD operations, query-method generation from method names, pagination, and custom JPQL/native queries all come from extending the right base interface, not from hand-written DAO code.

public interface UserRepository extends JpaRepository<User, Long> {
    List<User> findByStatus(String status); // implementation generated by Spring -- no method body needed
}
CrudRepository vs JpaRepository

CrudRepository provides the minimal set — save, findById, findAll, delete — with no pagination, sorting, or batch operations. JpaRepository extends both CrudRepository and PagingAndSortingRepository, adding pagination, sorting, batch operations, and JPA-specific lifecycle methods like flush()/saveAndFlush().

JpaRepository = CrudRepository + PagingAndSortingRepository + JPA-specific extras

⚠️ Pitfall: defaulting to JpaRepository "just in case" is fine in practice, but knowing the actual composition (not just "JpaRepository has more stuff") is what a senior-level answer demonstrates.

Writing Custom Queries

JPQL (entity-oriented, database-independent):

@Query("select u from User u where u.status = :status")
List<User> findByStatus(@Param("status") String status);

Native SQL (for cases JPQL can't efficiently express):

@Query(value = "select * from users where status = ?", nativeQuery = true)
List<User> findByStatusNative(String status);

@Modifying — required whenever a @Query writes rather than reads. Spring Data assumes every @Query is a SELECT by default; UPDATE/DELETE JPQL needs the explicit flag plus @Transactional:

@Modifying
@Transactional
@Query("update User u set u.status = :status where u.id = :id")
int updateUserStatus(Long id, String status);

Omitting @Modifying on an update/delete query throws an exception at execution time — not a compile error, which is exactly why it's an easy thing to miss until the query actually runs in a test or in production.

@Query delete vs deleteById() — a real behavioral difference, not just style: a @Modifying @Query("delete from User u where ...") executes a single bulk JPQL delete directly against the database — entities are never loaded into the persistence context, and JPA lifecycle callbacks (@PreRemove) never fire. deleteById() is a derived method that first fetches the entity via a SELECT, then deletes it — multiple SQL round-trips, lifecycle callbacks do fire, and it's slower. Use the bulk @Query form when performance matters and lifecycle callbacks aren't needed; use deleteById() when they are.

Custom Repository Implementation (When Derived Methods Aren't Enough)
public interface UserRepositoryCustom {
    List<User> findActiveUsers();
}

@Repository
public class UserRepositoryImpl implements UserRepositoryCustom {
    @PersistenceContext private EntityManager em;
    @Override public List<User> findActiveUsers() {
        return em.createQuery("select u from User u where u.active = true", User.class).getResultList();
    }
}

public interface UserRepository extends JpaRepository<User, Long>, UserRepositoryCustom { }

Use this specifically when you need raw EntityManager control (dynamic Criteria queries, complex native SQL assembly) that a derived query method or @Query annotation can't express cleanly.

findById() vs the Deprecated getOne()

findById() hits the database immediately, returns Optional<T>, and is safe to call outside a transaction — no proxy involved. getOne() returns a lazy proxy with no immediate database call; the actual query fires only when a property is first accessed, which means it must be called inside an active transaction, or accessing it later throws LazyInitializationException. Because of that footgun, getOne() is deprecated in modern Spring Data JPA, replaced by getReferenceById() (same lazy-proxy behavior, clearer name and contract).

⚠️ Pitfall: reaching for getOne()/getReferenceById() "because it's cheaper" without checking you're inside a transaction is one of the most common LazyInitializationException sources in real codebases — it looks correct in a unit test (where the whole test method might be @Transactional) and then breaks the moment it's called from a context without an open transaction/session.

Pagination and Sorting
Pageable pageable = PageRequest.of(0, 20, Sort.by("createdDate").descending());
Page<User> page = userRepository.findAll(pageable);

List<User> sorted = userRepository.findByStatusOrderByCreatedDateDesc("ACTIVE"); // via method name
List<User> sortedExplicit = userRepository.findByStatus("ACTIVE", Sort.by(Sort.Direction.DESC, "createdDate")); // via Sort param

Key interfaces: Pageable (the request — page number, size, sort), Page<T> (the response — content plus total count/pages, one extra COUNT query under the hood).

Query method naming rules: must start with find/read/get/query; property names must match entity field names, not database column names; supports And/Or/Between/LessThan/GreaterThan/Like/In/IsNull as composable keywords (e.g. findByStatusAndType(String status, String type)).

Query By Example (QBE)
User probe = new User();
probe.setStatus("ACTIVE");
probe.setCountry("India");
Example<User> example = Example.of(probe); // non-null properties become the search criteria
List<User> users = userRepository.findAll(example);

Useful for dynamic search forms where the set of filters varies per request, without writing a custom query method for every possible combination.

@Embeddable / @Embedded — Value Objects Without Their Own Table
@Embeddable
public class Address {
    private String street, city, state, zip;
}

@Entity
public class Employee {
    @Id private int id;
    private String name;
    @Embedded private Address address; // columns land in the SAME table as Employee, prefixed "address_"
}

Customizing column names:

@Embedded
@AttributeOverrides({
    @AttributeOverride(name = "street", column = @Column(name = "emp_street")),
    @AttributeOverride(name = "city", column = @Column(name = "emp_city"))
})
private Address address;

@Embeddable marks a class as a value type with no identity/table of its own; @Embedded tells JPA to inline that value type's fields as columns of the enclosing entity's own table — no join, no separate table, no foreign key.

Optimistic Locking — the Rules That Actually Get Checked
@Entity
public class Student {
    @Id private Long id;
    private String name;
    @Version private Integer version; // JPA increments this automatically on every update
}

The actual rules for the version attribute — asked precisely because they're easy to get subtly wrong: exactly one version attribute per entity; it must be on the primary table if the entity spans several tables; its type must be one of int/Integer/long/Long/short/Short/java.sql.Timestamp. You can read the version via the entity, but you must never manually update or increment it — only the persistence provider is allowed to, or the consistency guarantee it exists to provide breaks.

What actually happens on conflict: every transaction reading the entity captures the version value it saw; on update, JPA checks the version in the database still matches what was read — if another transaction updated it in between, the version no longer matches, and JPA throws OptimisticLockException, which the caller must catch and typically resolve via retry or surfacing a conflict to the user.

Pessimistic Locking — When the Lock Is Actually Held
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select a from Account a where a.id = :id")
Account findByIdForUpdate(Long id);

@Transactional
public void transferMoney(Long fromId, Long toId) {
    Account from = accountRepo.findByIdForUpdate(fromId); // 🔒 LOCK ACQUIRED HERE
    Account to   = accountRepo.findByIdForUpdate(toId);   // 🔒 LOCK ACQUIRED HERE
    from.debit(100);
    to.credit(100);
} // lock released when the transaction commits or rolls back

The generated SQL is effectively SELECT * FROM account WHERE id = ? FOR UPDATE — the lock is genuinely held at the database level, blocking any other transaction from reading (for update) or writing that row, from the moment the query executes until the enclosing transaction commits or rolls back.

⚠️ Pitfall — the lock WILL NOT happen if: the method isn't actually running inside an active @Transactional context (no transaction, no lock held past the single query), or the underlying database doesn't support the requested lock mode the way you'd expect. "Adding @Lock" without a genuine surrounding transaction is a common false sense of safety — always verify the method is inside @Transactional before trusting the lock is real.

Optimistic vs Pessimistic — when to actually choose which: pessimistic assumes conflicts are likely and blocks up front (bank balances, inventory counts, seat booking — anything where a lost update is expensive and retries are costly); optimistic assumes conflicts are rare and only detects them at commit time (user profiles, catalog data — anything mostly independent, where blocking readers/writers unnecessarily would hurt throughput for no real benefit).

Cascading a OneToMany Relationship — the Employee/Address Pattern

Requirement: saving an Employee should save its Address records; deleting an Employee should delete its addresses too.

class Employee {
    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    @JoinColumn(name = "emp_id")
    private List<Address> address = new ArrayList<>();
}

CascadeType.ALL propagates persist/merge/remove/refresh operations from Employee to its Address children automatically. orphanRemoval = true additionally deletes an Address the moment it's removed from the collection, even if the parent Employee itself isn't being deleted — a distinct, narrower guarantee than cascade delete, worth stating explicitly since the two are often conflated.

Finding employee count by postal code (a realistic query-method question building on the mapping above):

@Query("select count(e) from Employee e join e.address a where a.postCode = :postCode")
long countByPostalCode(@Param("postCode") String postCode);
Query Performance Optimization — the Real Toolkit

Projections — fetch only the fields you need, avoiding the full entity hydration cost:

public interface UserNameOnly { String getFirstName(); String getLastName(); }
public interface UserRepository extends JpaRepository<User, Long> {
    List<UserNameOnly> findByLastName(String lastName);
}

Fetch strategy control — FetchType.LAZY avoids loading associations until actually accessed:

@OneToMany(fetch = FetchType.LAZY, mappedBy = "user")
private List<Post> posts;

Batch fetching — reduces N+1-style query counts for collections:

@BatchSize(size = 10)
@OneToMany(fetch = FetchType.LAZY, mappedBy = "user")
private List<Post> posts;

Query caching for repeatedly executed, rarely-changing queries:

@QueryHints({ @QueryHint(name = "org.hibernate.cacheable", value = "true") })
@Query("SELECT u FROM User u WHERE u.lastName = :lastName")
List<User> findByLastName(@Param("lastName") String lastName);

Native queries for complex cases JPQL handles awkwardly, and Entity Graphs for defining a precise, per-query fetch plan without changing the entity's default fetch type globally.

⚠️ Pitfall: reaching for FetchType.EAGER by default "to avoid LazyInitializationException" is a common junior mistake that becomes a senior-level red flag — it silently turns every load of that entity into a bigger query (or an N+1 problem, see Part 2), regardless of whether the calling code actually needs the association. Default to LAZY, and fetch eagerly per query (via JOIN FETCH, @EntityGraph, or a projection) exactly when a specific use case actually needs it.

Spring Data JPA Versioning

Spring Boot 2.x ships with Spring Data JPA 2.x; Spring Boot 3.x ships with Spring Data JPA 3.x, which moved from javax.persistence to jakarta.persistence — a namespace change (not just a version bump) that breaks direct imports when upgrading, and is a common real-world migration pain point worth knowing by name.

Interview Q&A

Q: CrudRepository vs JpaRepository — what's the actual composition, not just "one has more features"? JpaRepository extends both CrudRepository and PagingAndSortingRepository, plus adds JPA-specific methods like flush()/saveAndFlush() — it's a composition, not a single bigger interface built from scratch.

Q: Why is getOne() deprecated, and what replaced it? It returns a lazy proxy with no immediate query, meaning it must be accessed inside an active transaction or it throws LazyInitializationException later — a common, confusing footgun. getReferenceById() replaced it with the same lazy semantics but a name that doesn't imply "already loaded."

Q: @Query-based bulk delete vs deleteById() — what's the actual behavioral difference? The bulk @Query delete executes a single JPQL statement directly against the database, skipping the persistence context and lifecycle callbacks entirely — fast, but @PreRemove never fires. deleteById() loads the entity first (a SELECT), then deletes it, firing lifecycle callbacks at the cost of extra round-trips.

Q: When exactly is a pessimistic lock actually held, and what's the most common false-safety mistake with it? It's held at the database level from the moment the locking query executes until the enclosing transaction commits or rolls back — but only if a real transaction is active. Adding @Lock to a repository method that isn't actually invoked inside @Transactional gives no real locking guarantee at all, which is the trap.

Q: CascadeType.ALL vs orphanRemoval = true on a @OneToMany — are they the same thing? No. CascadeType.ALL propagates persist/merge/remove operations from parent to children generally (e.g., deleting the parent deletes the children). orphanRemoval = true is narrower and different: it deletes a child the moment it's removed from the parent's collection, even if the parent itself is never deleted.

Hibernate Setup in Spring Boot

Mostly auto-configured — add spring-boot-starter-data-jpa, configure the datasource, and Spring Boot wires up EntityManagerFactory, Hibernate as the JPA provider, and transaction management with no manual SessionFactory or XML required.

spring.datasource.url=jdbc:mysql://localhost:3306/appdb
spring.datasource.username=root
spring.datasource.password=pass
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.database-platform=org.hibernate.dialect.MySQL8Dialect
session.get() vs session.load()

get() hits the database immediately and returns null if the record doesn't exist — always a fully initialized object, appropriate when you're not sure the entity exists and need to touch its data right away. load() returns a proxy with no immediate query — Hibernate defers the actual SELECT until a property is first accessed, and if the record turns out not to exist, accessing the proxy throws ObjectNotFoundException rather than returning null. Use load() when you're confident the entity exists and want to benefit from lazy loading (e.g., you only need the ID to set up an association, and never actually touch any other field).

⚠️ Pitfall: this is the exact same underlying trade-off as findById() vs getReferenceById() in Spring Data JPA (Part 1) — Hibernate's Session API and Spring Data's repository API expose the same two behaviors under different names, and interviewers sometimes ask both to see if you recognize they're the same concept.

HQL (Hibernate Query Language)

An object-oriented query language operating on entity names and entity fields, not table/column names — database-independent, and supports polymorphic queries (querying a superclass and getting results across all subclasses).

from Employee e where e.department.name = :dept

Case sensitivity, a genuinely easy trap: keywords (FROM, SELECT, WHERE) are not case-sensitive, but entity and property names are case-sensitive (they must match your Java class/field names exactly, not your database column names).

Commonly used Query methods: list() returns all results; executeUpdate() runs an update/delete; setFirstResult(n)/setMaxResults(n) implement pagination manually; setParameter(position/name, value) binds parameters safely (see SQL injection section below).

Associations and Their Default Fetch Types
| Association  | Default Fetch Type |
|---------------|---------------------|
| @ManyToOne    | EAGER               |
| @OneToMany    | LAZY                |
| @OneToOne     | EAGER               |
| @ManyToMany   | LAZY                |

⚠️ Pitfall — this table is asked precisely because the defaults are counter-intuitive to some: @ManyToOne and @OneToOne default to EAGER, while @OneToMany/@ManyToMany default to LAZY — the "to-one" side loads eagerly by default, the "to-many" side doesn't. Overriding @ManyToOne to LAZY explicitly is a common, deliberate performance optimization once a codebase matures, since the eager default can silently pull in more data than a given query path actually needs.

Bidirectional relationships and JSON infinite recursion: a bidirectional @OneToOne/@OneToMany means each side holds a reference back to the other — serializing either side to JSON naively recurses forever (Player → PlayerProfile → Player → ...). Fix with @JsonManagedReference on the owning side and @JsonBackReference on the inverse side — Jackson serializes only the managed side, breaking the cycle.

class PlayerProfile {
    @OneToOne(mappedBy = "playerProfile")
    @JsonBackReference // this side is NOT serialized, breaking the cycle
    private Player player;
}
The N+1 Problem

What it is: one query fetches N parent entities; then, because an association is lazily loaded and accessed per-parent in a loop, N additional queries fire — one per parent, to fetch each one's association separately. Example: 1 query for users, then N queries (one per user) for each user's orders.

Fixes, in order of how often they're reached for: 1. JOIN FETCH in JPQL — pulls the association into the same query. 2. @EntityGraph — declares a fetch plan for a specific query without changing the entity's global fetch type. 3. @BatchSize — batches the N follow-up queries into fewer, larger IN (...) queries instead of one-per-parent.

⚠️ Pitfall: N+1 is invisible in code review — the code looks correct, iterating a list and calling a getter. It only shows up as a real performance problem under load, in query logs or an APM tool, which is exactly why enabling spring.jpa.show-sql=true (or a proper SQL logging/monitoring setup) in staging/pre-prod is worth the noise — it's often the fastest way to actually see an N+1 pattern before a user reports slowness in production.

Criteria API — Dynamic, Type-Safe Queries
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
Root<Employee> emp = cq.from(Employee.class);
cq.where(cb.like(emp.get("fullName"), "%Hib%"));
List<Employee> results = em.createQuery(cq).getResultList();

Reach for Criteria specifically when the query's shape itself varies at runtime (an arbitrary combination of optional filters from a search form) — JPQL string-building for that scenario gets unwieldy and loses compile-time type checking fast.

Entity States

Transient — a new-ed object with no session association and no database representation; if abandoned, Hibernate simply ignores it. Persistent — associated with an active session; Hibernate tracks changes and synchronizes them to the database on flush/commit; entered via save()/persist(), or by loading via get()/load(). Detached — was persistent, but the session closed (or the object was explicitly evicted); no longer tracked, so further changes aren't automatically saved. Removed — marked for deletion, pending flush.

Reattaching a detached entity:

Entity managed = session.merge(detachedEntity); // returns a NEW managed instance -- the original stays detached
session.update(detachedEntity);                  // reattaches the SAME instance -- assumes it's not already persistent elsewhere

⚠️ Pitfall: merge() returns a different object than the one you passed in — continuing to use the original detached reference after calling merge() (instead of the returned managed instance) is a classic subtle bug, since further changes to the original object are silently untracked.

Session vs SessionFactory

SessionFactory is a thread-safe, heavyweight object responsible for creating Sessions — created once per application (or once per database, in a multi-datasource setup). Session is a single-threaded, lightweight object providing the actual CRUD operations — typically created per request or per transaction, opened, used, and closed within that single unit of work, never shared across threads.

hibernate.dialect and @GeneratedValue

hibernate.dialect tells Hibernate which database-specific SQL variant to generate (org.hibernate.dialect.PostgreSQLDialect, MySQL8Dialect, etc.) — different databases have different SQL quirks Hibernate needs to account for.

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
  • AUTO — Hibernate picks a strategy based on the dialect.
  • IDENTITY — the database generates it (auto-increment column) — simple, but disables JDBC batch inserts for that entity (each insert needs its generated ID back before the next can proceed).
  • SEQUENCE — uses a database sequence — Hibernate can pre-fetch sequence values, making it batch-insert-friendly, unlike IDENTITY.
  • TABLE — a separate table simulates a sequence — portable across databases lacking real sequence support, but slower.
First-Level Cache vs Second-Level Cache

First-level — scoped to a single Session, enabled by default, cannot be disabled. Loading/saving an entity within a session caches it; requesting the same entity again in that same session returns the cached instance without hitting the database. Cleared when the session closes.

Second-level — scoped to the SessionFactory, shareable across multiple sessions, not enabled by default, requires an explicit cache provider (Ehcache, Infinispan) and marking entities cacheable. Loading the same Employee in two entirely separate sessions can be served from the second-level cache instead of hitting the database twice, provided it's configured and the entity is cacheable.

Inheritance Strategies
  • SINGLE_TABLE — the whole hierarchy in one table, with a discriminator column. Fastest for queries (no joins), but sparse/nullable columns for subclass-specific fields across unrelated subclasses.
  • JOINED — each class gets its own table, joined via foreign keys on query. Normalized, no wasted columns, but slower due to joins.
  • TABLE_PER_CLASS — each concrete class gets its own complete table (no shared parent table). Avoids joins and nulls, but querying across the whole hierarchy needs UNIONs, which is slow and awkward.
Cascade and Inverse (mappedBy) — Two Genuinely Different Concepts

Cascade controls which operations propagate from parent to child (save/update/delete).

@OneToMany(mappedBy = "department", cascade = CascadeType.ALL)
private List<Employee> employees;

CascadeType.PERSIST/MERGE/REMOVE/REFRESH/ALL — saving/deleting a Department correspondingly saves/deletes its Employees.

Inverse (mappedBy) controls which side owns the foreign key — a completely orthogonal concern from cascading.

@Entity class Employee {
    @ManyToOne @JoinColumn(name = "department_id") // OWNING side -- manages the actual FK column
    private Department department;
}
@Entity class Department {
    @OneToMany(mappedBy = "department") // INVERSE side -- a mirror, does NOT manage the FK
    private List<Employee> employees;
}

⚠️ Pitfall — the mistake this rule exists to prevent: if you set the association only on the inverse (Department.employees) side and never on the owning (Employee.department) side, nothing gets persisted to the foreign key column at all — Hibernate only writes the FK based on the owning side's state, regardless of what the inverse side's in-memory collection looks like. Always set the relationship from the owning side, or explicitly keep both sides in sync in application code.

hbm2ddl.auto — What Each Value Actually Does
  • create — drops and recreates every table on startup; all existing data lost.
  • create-drop — same as create, plus drops tables again on shutdown; useful for tests only.
  • update — updates the schema to match entity mappings, preserving existing data.
  • validate — checks the schema matches the mappings, throws if not, makes no changes.
  • none — Hibernate touches nothing.

⚠️ Pitfall: create/update in a real production environment is a well-known incident-waiting-to-happen — create destroys data outright, and even update can behave unexpectedly on some databases/versions for certain schema changes (column type narrowing, renames Hibernate can't safely infer). Production should run validate (or none), with schema changes managed through an explicit migration tool (Flyway/Liquibase) instead of Hibernate auto-DDL.

SQL Injection, final Entities, and Named Queries

SQL injection: Hibernate's query APIs (HQL/JPQL, Criteria, JPA Query) use parameter binding under the hood, compiling down to JDBC PreparedStatement parameters — user input passed via setParameter() is always treated as data, never as SQL syntax. The risk reappears specifically when native queries are built via raw string concatenation instead of bound parameters.

Why an entity class should never be final: Hibernate implements lazy loading by generating a runtime proxy subclass of the entity. A final class can't be subclassed — lazy loading (and by extension, a large part of Hibernate's performance model) silently breaks or falls back to eager loading for that entity.

Named queries — precompiled and validated at SessionFactory startup (catching typos/errors immediately at boot rather than at first execution), and centralized/easier to maintain for complex queries. Trade-off: not customizable at runtime beyond supplied parameters (can't change sorting or structure dynamically), and changing one requires reloading the SessionFactory — can't be hot-swapped in a running application.

Flush Time and JDBC Batching

"Flushing" means writing pending in-memory changes (INSERT/UPDATE/DELETE) to the actual database. Hibernate flushes: before a transaction commits (so the commit reflects the latest state), before executing a query that could be affected by pending changes (so the query sees consistent data), and whenever explicitly triggered via session.flush().

hibernate.jdbc.batch_size controls how many statements of the same type get grouped into a single JDBC batch instead of executing individually: 1. Hibernate collects same-type SQL statements (e.g., all pending Employee inserts) in memory. 2. Adds them to an internal JDBC batch. 3. Once the configured batch size is reached, executes the whole batch in one round-trip. 4. A flush/commit before the batch fills executes whatever's accumulated immediately, even partially filled.

batch_size = 0 (or unset) means no batching — every statement executes individually, increasing database round-trip overhead; acceptable for small operations or when debugging (individual statements are easier to see in logs), but a real throughput cost at scale. Note also that GenerationType.IDENTITY (above) effectively disables batching for inserts on that entity, regardless of the configured batch size, since each row's generated key must come back before the next insert can be batched.

Interview Q&A

Q: session.get() vs session.load() — what's the actual difference, and when would you pick each? get() hits the database immediately and returns null if missing; load() returns a lazy proxy that only queries on first property access and throws ObjectNotFoundException if the row turns out not to exist. Use get() when existence is uncertain and you need the data now; load() when you're confident it exists and want to defer the actual query.

Q: What's the N+1 problem, and what are the three standard fixes? One query for parent entities, then one additional query per parent for a lazily-loaded association accessed in a loop. Fix with JOIN FETCH (pull the association into the same query), @EntityGraph (declare a per-query fetch plan), or @BatchSize (batch the follow-up queries into fewer, larger ones).

Q: Cascade vs Inverse/mappedBy — are these the same concept? No — cascade controls which operations (save/delete/etc.) propagate from parent to child; mappedBy/inverse controls which side owns the foreign key in a bidirectional relationship. Setting a bidirectional association only from the inverse side, with the owning side never updated, persists nothing to the actual FK column.

Q: Why can't an entity class be final in Hibernate? Hibernate implements lazy loading via a runtime-generated proxy subclass of the entity — a final class can't be subclassed, so lazy loading breaks (or silently falls back to eager) for that entity.

Q: Why is hbm2ddl.auto=update risky in production, even though it doesn't drop tables like create does? It can behave unpredictably for schema changes Hibernate can't safely infer from entity mappings (column narrowing, renames), and it bypasses any real migration history/rollback story. Production should use validate/none with schema changes managed by an explicit migration tool like Flyway or Liquibase.
